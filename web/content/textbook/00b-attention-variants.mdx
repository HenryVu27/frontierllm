## Why attention variants exist {#intro}

The self-attention mechanism introduced in [the transformer block](/textbook/00a-transformer#attention-math) has a single structural property that dominates its engineering cost at inference: every token must store its key and value representations so that future tokens can attend to them. This storage is called the **KV cache**. Under the standard multi-head formulation, the KV cache grows linearly with sequence length and linearly with model depth — but the proportionality constant is large enough that it becomes the binding constraint long before compute does.

For a concrete sense of scale: a hypothetical Llama-3.1-8B model using unmodified multi-head attention with $H=32$ query heads, head dimension $d_k=128$, and $L=32$ layers would require approximately 67 GB of KV cache to serve a single 128K-token context in BF16. The model weights themselves occupy about 16 GB. The KV cache is four times heavier than the model.

The variants covered in this chapter — Multi-Query Attention (MQA), Grouped-Query Attention (GQA), and Multi-head Latent Attention (MLA) — are each a different answer to the question: *how much quality are we willing to trade for a given reduction in KV cache size?* The chapter follows the historical progression, then turns to three orthogonal but related topics: sliding-window attention (bounding the cache by sequence position rather than head count), FlashAttention (an algorithmic improvement that makes the quadratic forward pass tractable in HBM), and sub-quadratic alternatives including SSM-hybrid architectures.

<Callout variant="note" title="Notation conventions for this chapter">
Throughout this chapter: $d$ denotes the model's hidden dimension; $H$ the number of query heads; $G$ the number of KV heads (so $G=H$ is MHA and $G=1$ is MQA); $d_k = d_v = d/H$ the per-head key and value dimension; $L$ the number of transformer layers; $N$ the sequence length. BF16 is assumed for cache-size calculations (2 bytes per element). These match the conventions in [the transformer block chapter](/textbook/00a-transformer#attention-math).
</Callout>

## Multi-Head Attention {#mha}

### Mechanism

Multi-Head Attention (MHA), introduced in Vaswani et al. (2017), projects each token's hidden state $\mathbf{h} \in \mathbb{R}^d$ into $H$ independent query, key, and value triplets, runs scaled dot-product attention within each head, concatenates the outputs, and projects back to the residual stream dimension.

For head $i \in \{1, \ldots, H\}$:

$$\mathbf{head}_i = \text{Attention}(\mathbf{X}W_i^Q,\ \mathbf{X}W_i^K,\ \mathbf{X}W_i^V)$$

$$\text{Attention}(Q, K, V) = \text{softmax}\!\left(\frac{QK^\top}{\sqrt{d_k}}\right)\!V$$

where the projection matrices $W_i^Q, W_i^K, W_i^V \in \mathbb{R}^{d \times d_k}$ with $d_k = d/H$, and the output projection $W^O \in \mathbb{R}^{Hd_v \times d}$.

The full MHA output is:

$$\text{MHA}(\mathbf{X}) = \bigl[\mathbf{head}_1;\, \ldots;\, \mathbf{head}_H\bigr]\, W^O$$

### Parameter count

The total learnable parameter count across all four weight matrices in one MHA layer is:

$$\underbrace{H \cdot d \cdot d_k}_{W^Q} + \underbrace{H \cdot d \cdot d_k}_{W^K} + \underbrace{H \cdot d \cdot d_v}_{W^V} + \underbrace{H d_v \cdot d}_{W^O} = 4d^2$$

since $H \cdot d_k = d$ and $H \cdot d_v = d$, each projection is effectively $d \times d$.

### KV cache per token

During autoregressive decoding, the model must retain the key and value tensors for all previously generated tokens. With $H$ heads, each storing a $d_k$-dimensional key and a $d_v$-dimensional value, the KV cache size per token per layer in BF16 is:

$$\text{KV bytes/token/layer} = 2 \times H \times d_k \times 2 = 4Hd_k = 4d$$

For a full model with $L$ layers: $4dL$ bytes per token.

### Worked example: Llama-3.1-8B (hypothetical MHA)

The Llama-3.1-8B architecture has $d=4096$, $H=32$, $d_k=128$, $L=32$. The actual model uses GQA (discussed in the next section), but as a reference calculation, if it used MHA:

- KV bytes per token per layer: $4 \times 4096 = 16{,}384$ bytes = 16 KB
- KV bytes per token (full model): $16{,}384 \times 32 = 524{,}288$ bytes $\approx 512$ KB
- KV cache at 128K tokens: $512 \times 128{,}000 \approx 65$ GB

This 65 GB figure exceeds the 48 GB of an A100 GPU by itself, before the model weights. The actual model uses GQA with $G=8$ KV heads, bringing the cache to approximately 16 GB at 128K tokens — a factor-of-four reduction.

### Frontier practice and status

Pure MHA is no longer used in frontier decoder-only models as of 2025–2026. It remains in encoder models (BERT-family, T5 encoders) and some small-scale research settings. All major open-weight decoder models in 2024–2026 — Llama 3.x, Mistral, DeepSeek-V2/V3, Qwen-2.5, Gemma 2 — use either GQA or MLA. The memory pressure described above is the direct motivation.

Cross-reference: the base attention derivation, including the $\sqrt{d_k}$ scaling motivation, is in [the transformer block](/textbook/00a-transformer#attention-math). How RoPE is applied to queries and keys before the dot-product is covered in [positional encodings](/textbook/00c-positional-encodings#rope).

## Multi-Query Attention {#mqa}

### Mechanism

Multi-Query Attention (MQA), introduced by Shazeer (2019), keeps $H$ query heads but reduces to a *single* shared key head and a single shared value head across all queries. All query heads perform attention against the same $K$ and $V$ matrices.

<Sidenote>Shazeer's MQA paper (arXiv:1911.02150) was published in November 2019 but received little attention for three years. It became widely cited after the KV-cache bottleneck re-emerged at scale in 2022–2023. PaLM (Google, 2022) was among the first large-scale models to adopt it.</Sidenote>

The weight matrices become: $W^Q \in \mathbb{R}^{d \times d}$ (same as MHA), $W^K \in \mathbb{R}^{d \times d_k}$ (single head), $W^V \in \mathbb{R}^{d \times d_v}$ (single head). The output projection $W^O \in \mathbb{R}^{Hd_v \times d}$ remains unchanged.

### KV cache per token

With a single KV head, the cache per token per layer is:

$$\text{KV bytes/token/layer} = 2 \times d_k \times 2 = 4d_k = \frac{4d}{H}$$

This is a factor-$H$ reduction from MHA. For $H=32$: 32× smaller cache.

### Worked example: Llama-3.1-8B (hypothetical MQA)

Using the same parameters as above ($d=4096$, $H=32$, $d_k=128$, $L=32$) but with MQA:

- KV bytes per token per layer: $4 \times 128 = 512$ bytes
- KV cache at 128K tokens: $512 \times 32 \times 128{,}000 \approx 2.1$ GB

A 32× reduction from the MHA baseline. The cache is now negligible relative to the model weights.

### Quality tradeoff

Shazeer (2019) reported minor quality degradation for the sizes tested in that paper. Ainslie et al. (2023) later quantified this more carefully on T5-XXL: MQA achieves decode throughput of 0.24 s/step versus MHA's 1.51 s/step (a 6.3× speedup), but ROUGE-1 drops from 47.2 to 46.6 and TriviaQA F1 from 81.9 to 81.3 — small in absolute terms but measurable.

At larger scales, the quality gap with MHA grows. The representational bottleneck is structural: all query heads must attend through the same key and value representations, so different heads cannot develop independently specialised retrieval behaviour at the KV level.

### Frontier practice

MQA is used in PaLM (Google, 2022) and Falcon (Technology Innovation Institute, 2023). Most post-2023 frontier models prefer GQA, which recovers most of MHA's quality at nearly MQA-level speed, as shown next.

## Grouped-Query Attention {#gqa}

### Mechanism

GQA (Ainslie et al., 2023) generalises both MHA and MQA: divide the $H$ query heads into $G$ groups; all heads within a group share one key head and one value head. Setting $G=H$ recovers MHA; setting $G=1$ recovers MQA. In practice, $G$ is a hyperparameter chosen by the model designer.

Each group $g \in \{1,\ldots,G\}$ has dedicated matrices $W_g^K \in \mathbb{R}^{d \times d_k}$ and $W_g^V \in \mathbb{R}^{d \times d_v}$. Query heads within group $g$ (i.e., heads $\{(g-1)(H/G)+1,\ldots,g(H/G)\}$) read from the same $K_g, V_g$ representation.

### KV cache per token

With $G$ KV heads:

$$\text{KV bytes/token/layer} = 2G \cdot d_k \cdot 2 = 4G \cdot d_k = \frac{4Gd}{H}$$

Cache reduction relative to MHA: $H/G$. For $H=32$, $G=8$: a 4× reduction. For $H=64$, $G=8$: an 8× reduction.

### Worked example: Llama-3.1 family

The Llama-3.1 models (Meta, 2024) all use GQA with $G=8$ KV heads across three sizes, verified from public HuggingFace config files:

**Llama-3.1-8B** ($d=4096$, $H=32$, $G=8$, $d_k=128$, $L=32$):
- KV bytes per token per layer: $4 \times 8 \times 128 = 4{,}096$ bytes
- KV cache at 128K tokens: $4096 \times 32 \times 128{,}000 \approx 16$ GB
- Fits alongside the 16 GB model weights on a 48 GB GPU.

**Llama-3.3-70B** ($d=8192$, $H=64$, $G=8$, $d_k=128$, $L=80$):
- KV bytes per token per layer: $4 \times 8 \times 128 = 4{,}096$ bytes (same head arithmetic)
- KV cache at 128K tokens: $4096 \times 80 \times 128{,}000 \approx 42$ GB
- Feasible on an 8×A100 node alongside the ~140 GB model.

**Llama-3.1-405B** ($d=16384$, $H=128$, $G=8$, $d_k=128$, $L=126$):
- The Q:KV ratio is 16:1 — the most aggressive GQA among the Llama-3.1 family.
- KV bytes per token per layer: $4 \times 8 \times 128 = 4{,}096$ bytes
- KV cache at 128K tokens: $4096 \times 126 \times 128{,}000 \approx 66$ GB

All three models land in the same 4 KB per token per layer range for GQA with $G=8$, despite the 50× difference in total parameter count. This is because the KV cache cost is determined by $G \cdot d_k$, and all three models hold $G \cdot d_k = 8 \times 128 = 1024$ constant.

### Quality: the uptraining recipe

Ainslie et al. (2023) showed that existing MHA checkpoints can be converted to GQA without full re-training. The conversion procedure mean-pools the $H/G$ original K/V head projections within each group, producing a starting point close to a well-trained GQA model. Re-training for approximately 5% of the original pre-training compute (around 600 TPUv3 chip-days for T5-XXL in their experiments) recovers the converted checkpoint to near-MHA quality.

The optimal value of $G$ is a function of model scale, batch size, and target context length, and there is no universal formula for it. ## Multi-head Latent Attention {#mla}

### Mechanism

MLA (DeepSeek-AI, 2024) takes a different approach to the KV cache problem. Rather than reducing the *number* of KV heads, MLA reduces the *dimensionality* of what is cached per token. The key idea is a low-rank bottleneck: keys and values are compressed into a shared latent vector before caching, then decompressed on-the-fly for attention. An "absorption trick" at inference time eliminates the decompression pass entirely.

MLA was introduced in DeepSeek-V2 (May 2024) and retained with refinements in DeepSeek-V3 (December 2024). No other major open-weight model family had adopted MLA as of early 2026, though the technique has attracted substantial research interest.

### Math: the compression path

Let $\mathbf{h}_t \in \mathbb{R}^d$ be the hidden state at position $t$, $n_h$ be the number of attention heads, and $d_h$ the per-head dimension.

Standard KV projection would compute:

$$[\mathbf{k}_t;\, \mathbf{v}_t] = W^{KV}\mathbf{h}_t, \quad W^{KV} \in \mathbb{R}^{2n_h d_h \times d}$$

MLA instead factorises this through a low-rank latent:

$$\mathbf{c}_t^{KV} = W^{DKV}\mathbf{h}_t, \quad W^{DKV} \in \mathbb{R}^{d_c \times d}, \quad \mathbf{c}_t^{KV} \in \mathbb{R}^{d_c}$$

where $d_c \ll 2n_h d_h$ is the *KV compression dimension*. This latent vector $\mathbf{c}_t^{KV}$ is what gets cached — a $d_c$-dimensional vector rather than a $2n_h d_h$-dimensional one.

Full-rank keys and values are recovered on the fly:

$$[\mathbf{k}_{t,1}^C;\, \ldots;\, \mathbf{k}_{t,n_h}^C] = W^{UK}\mathbf{c}_t^{KV}, \quad W^{UK} \in \mathbb{R}^{n_h d_h \times d_c}$$
$$[\mathbf{v}_{t,1};\, \ldots;\, \mathbf{v}_{t,n_h}] = W^{UV}\mathbf{c}_t^{KV}, \quad W^{UV} \in \mathbb{R}^{n_h d_h \times d_c}$$

An analogous compression is applied on the query side to a latent $\mathbf{c}_t^Q \in \mathbb{R}^{d_c'}$ using $W^{DQ} \in \mathbb{R}^{d_c' \times d}$ and expanded via $W^{UQ}$. Query latents are not cached (queries are regenerated at each decode step), so $d_c'$ does not affect the cache size.

### Decoupled RoPE

Rotary positional encoding (RoPE) cannot be applied to the compressed latent $\mathbf{c}_t^{KV}$ before caching, because the rotation is position-dependent and the absorption trick (see below) assumes a fixed decompression matrix. MLA resolves this with a separate "decoupled" RoPE key path:

$$\mathbf{k}_t^R = \text{RoPE}(W^{KR}\mathbf{h}_t)$$

This RoPE key $\mathbf{k}_t^R \in \mathbb{R}^{d_h^R}$ is cached alongside $\mathbf{c}_t^{KV}$ and concatenated per head:

$$\mathbf{k}_{t,i} = [\mathbf{k}_{t,i}^C;\, \mathbf{k}_t^R]$$

The full treatment of this RoPE interaction is in [positional encodings](/textbook/00c-positional-encodings#rope). The practical consequence is that MLA caches two things per token per layer: the latent $\mathbf{c}_t^{KV}$ and the decoupled RoPE key $\mathbf{k}_t^R$.

### Absorption trick

At inference, the decompression matrices $W^{UK}$ and $W^{UV}$ can be absorbed into the query and output projections respectively, so the forward pass works entirely in the latent space without an explicit decompression step.

The attention score between query $\mathbf{q}_{t,i}$ and a cached latent $\mathbf{c}_s^{KV}$ (for position $s$) can be written:

$$\mathbf{q}_{t,i}^\top \mathbf{k}_{s,i}^C = \mathbf{q}_{t,i}^\top W^{UK}_{(i)} \mathbf{c}_s^{KV}$$

Because $(W^{UK}_{(i)})^\top W^{UK}_{(i)}$ is a constant matrix (independent of $s$), its effect can be absorbed into the query projection at model-load time. Similarly, $W^{UV}$ is absorbed into the output projection $W^O$. The result: during a decode step, only the cached $\mathbf{c}_t^{KV}$ and $\mathbf{k}_t^R$ tensors need to be read from HBM, not reconstructed per-head keys and values.

<Callout variant="warning" title="Parameter count: MLA is not simpler than MHA">
MLA reduces the KV cache at inference, but it adds parameters. The down-projection matrices $W^{DKV}$ and $W^{DQ}$ and up-projection matrices $W^{UK}$, $W^{UV}$, $W^{UQ}$ are new weights that MHA does not have. MLA is not cheaper to train than MHA, and a complete parameter-count comparison requires working through the full set of matrices with concrete dimensions. The benefit is entirely at inference, where the smaller cached latent directly reduces memory bandwidth.
</Callout>

### DeepSeek-V3 hyperparameters and KV cache numbers

DeepSeek-V3 (arXiv:2412.19437, December 2024) uses the following MLA configuration:

- $d = 7168$ (embedding dimension)
- $n_h = 128$ (attention heads)
- $d_h = 128$ (per-head dimension)
- $d_c = 512$ (KV compression dimension, **this is what is cached**)
- $d_c' = 1536$ (query compression dimension, not cached)
- $d_h^R = 64$ (decoupled RoPE dimension, **also cached**)

KV cache per token per layer in BF16:

$$\text{MLA: } (d_c + d_h^R) \times 2 = (512 + 64) \times 2 = 1{,}152 \text{ bytes}$$

For comparison, the MHA-equivalent (same $n_h=128$, $d_h=128$, K+V):

$$\text{MHA: } 2 \times 128 \times 128 \times 2 = 65{,}536 \text{ bytes}$$

MLA is approximately 57× smaller than the MHA equivalent for the same architecture. Compared to a GQA-8 baseline with the same $n_h=128$, $d_h=128$ (e.g., Llama-3.1-405B):

$$\text{GQA-8: } 2 \times 8 \times 128 \times 2 = 4{,}096 \text{ bytes}$$

MLA is approximately 3.6× smaller than GQA-8 at this scale. In absolute terms for DeepSeek-V3's 61-layer architecture at 128K context: MLA caches approximately 7.6 GB versus an estimated 213.5 GB for the MHA equivalent (per the DeepSeek-V3 technical report, December 2024). McCormick (2025) reports per-model full-context cache sizes as approximately 70 KB/token for DeepSeek-V3 MLA versus approximately 516 KB/token for Llama-3.1-405B GQA-8 — a roughly 7× advantage for MLA even relative to GQA at that scale.

### Quality

The DeepSeek-V2 report (2024) notes that MLA "slightly exceeds the MHA baseline in some comparisons" in their internal ablations. Independent small-scale experiments (Planetbanatt, 2025) show a small quality gap between MLA with RoPE and MHA at 35M-parameter scale on WikiText (96.70 vs 94.31 perplexity), which the author notes likely closes at larger scale. The weight of current evidence suggests MLA does not substantially degrade quality relative to MHA; the tradeoff is primarily about training-time complexity versus inference-time cache efficiency.

Some MLA implementations add QK-norm to stabilise attention logits; the full treatment of that technique is in [normalisation and initialisation](/textbook/00d-normalization-and-init#qk-norm).

## Sliding-Window Attention {#sliding-window}

### Mechanism

Sliding-window attention (SWA) bounds the KV cache by *position* rather than by the number of heads. Each token attends only to the $W$ most recent tokens; all attention to positions further back is masked to $-\infty$ before the softmax. This changes the cache from $O(N)$ to $O(W)$ — fixed size regardless of sequence length.

The idea predates Mistral 7B: Longformer (Beltagy et al., 2020) and BigBird (Zaheer et al., 2020) both use local attention windows as components of their sparse attention designs. Mistral 7B (Jiang et al., 2023) popularised SWA for decoder-only generation models.

### Math

For a query at position $i$, attention is restricted to keys at positions $\{j : i - W \le j \le i\}$. The attention mask sets $S_{ij} = -\infty$ for $j < i - W$.

Through the depth of the network, each token's *receptive field* compounds: at layer $k$, the effective attention span is $k \times W$ positions. For Mistral 7B with $W=4096$ and $L=32$ layers, the theoretical maximum span is $32 \times 4096 \approx 131$K tokens — though recall of distant tokens degrades in practice because the information must be relayed through intermediate representations, not through direct attention.

### Rolling buffer KV cache

The standard implementation uses a **rolling buffer** with modular indexing: the KV cache has exactly $W$ slots per head. The key and value tensors for position $i$ are stored in slot $i \bmod W$. When position $i + W$ is processed, its entry overwrites position $i$'s slot, since position $i$ is now outside the window.

This gives a constant $O(W)$ memory footprint at all sequence lengths. The KV cache size is known at model-load time and never grows — a practical advantage for memory budgeting in inference systems.

The interaction with positional encodings requires care: at positions beyond $W$, two distinct sequence positions map to the same cache slot. Absolute position indices are still passed to RoPE, so the rotation correctly distinguishes them; the masking logic ensures that stale (overwritten) slot entries are never attended to. Full details of this interaction are in [positional encodings](/textbook/00c-positional-encodings#rope).

### Hybrid local+global attention

A single SWA layer cannot attend to positions beyond distance $W$. One design response is to alternate between local SWA layers and full-attention ("global") layers at regular intervals, so that global layers maintain long-range recall while local layers handle most of the computation cheaply.

**Gemma 2** (Google, arXiv:2408.00118, August 2024) implements this pattern: sliding-window layers with $W=4096$ alternate with global attention layers with $W=8192$ in a 1:1 ratio throughout the network. All layers use GQA with $G=2$. The paper reports that Gemma 2-27B achieves higher Chatbot Arena Elo (1218) than Llama-3 70B (1206) despite its smaller size, attributing part of this to the local/global attention design.

**Mistral 7B** (Jiang et al., 2023) uses SWA with $W=4096$ and GQA throughout all layers. The rolling buffer enables inference at sequences far beyond the training context without the KV cache growing.

**RecurrentGemma / Griffin** (Google DeepMind, 2024) uses local sliding-window MQA windows interleaved with gated linear recurrences — part of the hybrid SSM-attention family covered in the final section.

<Callout variant="warning" title="Long-distance recall limitations">
Tokens at distance $> W$ from the current position are invisible within any single SWA layer. While layer-stacking extends the theoretical span to $L \times W$, the information relay through intermediate representations is imperfect. Perplexity on documents longer than the effective receptive field degrades measurably. In practice, Mistral 7B performs well on tasks that don't require long-range exact recall (e.g., local reasoning, code completion) but below full-attention models on tasks requiring retrieval from the far context.
</Callout>

## FlashAttention {#flash-attention}

### The problem: HBM bandwidth is the bottleneck

Standard attention for a sequence of length $N$ requires materialising the attention score matrix $S = QK^\top / \sqrt{d_k} \in \mathbb{R}^{N \times N}$ in high-bandwidth memory (HBM) on the GPU. The sequence of operations is:

1. Compute $S = QK^\top / \sqrt{d_k}$ — write $N^2$ floats to HBM.
2. Compute $P = \text{softmax}(S)$ — read $N^2$ floats from HBM, write $N^2$ back.
3. Compute $O = PV$ — read $N^2 + Nd$ floats from HBM.

Total HBM I/O: $\Theta(N^2)$ reads and writes.

For $N=8192$ and $d_k=128$ in FP16, the score matrix $S$ alone occupies $8192^2 \times 2 = 134$ MB *per head*. An A100 GPU has approximately 80 GB of HBM and 2 TB/s of HBM bandwidth, but only $\sim$20 MB of on-chip SRAM (shared memory + L2 cache) and orders of magnitude more SRAM bandwidth. Standard attention is therefore **memory-bandwidth bound**, not compute bound: the GPU spends most of its time waiting for HBM reads and writes, not performing floating-point operations.

FlashAttention (Dao et al., NeurIPS 2022) is an exact attention algorithm — computing the same result as standard attention, with no approximation — that eliminates the $\Theta(N^2)$ HBM traffic.

### The online softmax recurrence

The central obstacle to avoiding the $N \times N$ matrix is the softmax: normalising each row requires knowing the maximum and sum across the entire row. Computing them block-by-block requires maintaining running statistics.

Milakov and Gimelshein (2018) showed that a numerically stable running softmax can be maintained with three statistics: a running maximum $m$, a running denominator $\ell$, and a running output accumulator $O$. For each new block of keys and values, denoted $K_j, V_j$, let $m_j$ and $\ell_j$ be the local max and sum-of-exponentials for that block. The update rule is:

$$m_{\text{new}} = \max(m,\, m_j)$$

$$\ell_{\text{new}} = e^{m - m_{\text{new}}} \cdot \ell + e^{m_j - m_{\text{new}}} \cdot \ell_j$$

$$O_{\text{new}} = \frac{e^{m - m_{\text{new}}} \cdot \ell \cdot O + e^{m_j - m_{\text{new}}} \cdot \ell_j \cdot O_j}{\ell_{\text{new}}}$$

After processing all blocks, the accumulated $O$ is identical — to floating-point precision — to $\text{softmax}(QK^\top)V$ computed by the standard two-pass algorithm. The formal equivalence proof is in Appendix B of Dao et al. (2022), building on Milakov and Gimelshein (2018).

<Sidenote>The online softmax recurrence is the mathematical foundation of FlashAttention, but it was not developed by Dao et al. — it appeared in Milakov and Gimelshein (2018) as a numerically stable algorithm for GPU softmax kernels. The contribution of FlashAttention is using this recurrence to design an IO-aware tiled attention algorithm.</Sidenote>

### IO-aware tiling algorithm

The online recurrence enables a tiling strategy that keeps all active data in SRAM rather than staging it through HBM. Conceptually, for each block of queries $Q_i$:

1. Load $Q_i$ into SRAM from HBM (one read).
2. For each block of keys and values $K_j, V_j$: load into SRAM, accumulate into the running $m$, $\ell$, $O$ statistics using the recurrence above, then discard from SRAM.
3. Write the completed output block $O_i$ to HBM (one write).

Each element of $Q$, $K$, $V$ is read from HBM exactly once. The output $O$ is written to HBM exactly once. Total HBM I/O: $\Theta(Nd)$ rather than $\Theta(N^2)$.

The arithmetic work is unchanged: FlashAttention still computes all $N^2 d$ floating-point operations that standard attention requires. It does not reduce FLOPs; it reduces memory transfers. The speedup comes from the fact that, at typical LLM sequence lengths ($N \ge 2048$), the operation is memory-bandwidth bound, so reducing HBM traffic directly reduces wall-clock time.

**Constraint:** the tile size is bounded by the SRAM capacity. Specifically, one tile of $Q_i$ and one tile of $K_j, V_j$ must fit simultaneously in SRAM. For an A100 with $\sim$20 MB of usable SRAM and FP16, this limits the tile to a few thousand tokens per head — well within the range of typical sequence lengths.

FlashAttention-1 (2022) reported 2–4× wall-clock speedup over PyTorch standard attention at the time, and 15–17% end-to-end speedup on GPT-2 training (1K tokens). FlashAttention-2 (Dao, ICLR 2024) focused on reducing the fraction of operations that are not matrix multiplications (which GPUs handle less efficiently than matmuls), and on better parallelism across attention heads and batch dimensions. FlashAttention-3 (Shah et al., NeurIPS 2024) exploits features specific to the Hopper GPU microarchitecture (H100/H200): asynchronous tensor memory access (TMA), warp specialisation to overlap data movement with computation, and an FP8 computation mode that uses incoherent processing (random Hadamard transform before quantisation) to spread activation outliers and reduce quantisation error.

### Frontier practice

FlashAttention-2 is the de facto standard attention kernel across major training frameworks as of 2024: it is integrated into PyTorch 2.x via `torch.nn.functional.scaled_dot_product_attention`, JAX via Pallas, NeMo, and DeepSpeed. FlashAttention-3 is used in H100 clusters. Adaptations are required for non-standard attention patterns: SWA (mask the tiling loop), causal attention (triangular mask applied per tile), and MLA (a custom absorption path is needed). AMD MI300X support for MLA-adapted FlashAttention was added in 2025.

<Callout variant="info" title="What FlashAttention does NOT do">
FlashAttention does not reduce the quadratic FLOPs of attention — it computes the same arithmetic, faster. It does not enable attention to longer sequences by reducing computation; it enables them by fitting the computation into available memory bandwidth. For truly sub-quadratic attention complexity, different architectures (linear attention, SSMs) are required.
</Callout>

## Linear and sub-quadratic alternatives {#linear-attention}

### Linear attention: the kernel trick

Katharopoulos et al. (ICML 2020) observed that if the softmax is replaced by a positive-definite kernel function $\phi$, the attention operation factorises:

$$\text{Attention}(Q, K, V)_i = \frac{\phi(\mathbf{q}_i)^\top \bigl(\sum_j \phi(\mathbf{k}_j) \mathbf{v}_j^\top\bigr)}{\phi(\mathbf{q}_i)^\top \sum_j \phi(\mathbf{k}_j)}$$

The matrix $\sum_j \phi(\mathbf{k}_j) \mathbf{v}_j^\top \in \mathbb{R}^{d \times d}$ can be computed incrementally. At inference this becomes an RNN recurrence with $O(d^2)$ state and $O(1)$ per-step cost, rather than attention over all prior tokens. Training is $O(Nd^2)$ rather than $O(N^2 d)$, sub-quadratic in $N$ when $N > d$.

The quality cost of replacing softmax with a kernel is substantial in practice. On standard language modelling benchmarks, linear attention models trained at 125M parameters show perplexity gaps of 3–5 points relative to softmax transformers on WikiText-103. The kernel approximation cannot faithfully represent the sharp, selective attention patterns that softmax produces on in-context learning tasks.

### RWKV

RWKV (Peng et al., EMNLP 2023) formulates a model that trains in parallel like a transformer (using an efficient custom CUDA operator, $O(N)$ wall-clock) but runs inference as an RNN ($O(1)$ per step, constant memory). The WKV operator provides a learned temporal decay: older context is exponentially down-weighted per a learned per-channel decay parameter. RWKV does not maintain a KV cache; it maintains a fixed-size hidden state that summarises all prior context.

RWKV was scaled to 14B parameters in 2023 — at that time the largest publicly trained dense RNN. The architecture has continued development (RWKV-7, "Goose", was released in 2024–2025) with an active open-source community. Performance is competitive with similarly-sized transformers on many tasks, with advantages in inference memory and latency at long contexts. In-context learning on tasks requiring exact retrieval from the context is measurably weaker than full-attention models.

### Hyena

Hyena (Poli et al., ICML 2023) replaces attention with implicitly parametrised long convolutions (Hyena filters) interleaved with element-wise gating. At sequence length 2K, Hyena achieves comparable quality to a standard transformer at approximately 20% lower training-compute; at 64K, it is 100× faster than optimised attention.

Hyena demonstrated transformer-quality perplexity on WikiText103 at ICML 2023 and has not, as of early 2026, been adopted in major frontier models.

### Mamba and Mamba-2

**Mamba** (Gu and Dao, ICLR 2024) is a Selective State Space Model (SSM). The key innovation over prior SSMs (S4) is making the SSM parameters $\Delta$, $B$, $C$ input-dependent, allowing the model to selectively propagate or discard information from the context. This breaks the fixed-convolution structure of S4 but Gu and Dao designed a hardware-aware parallel scan algorithm that maintains training efficiency. Mamba-3B matches transformer quality at 3B parameters and outperforms it on several language modelling benchmarks at the 2× larger scale.

**Mamba-2** (Gu and Dao, ICML 2024) establishes a *Structured State Space Duality* (SSD): SSMs and linear attention are dual views of the same semiseparable matrix structure. The SSD framework yields a "chunkwise" algorithm that is 2–8× faster than Mamba-1's selective scan for the same model. It also clarifies the theoretical relationship between transformers and SSMs, providing a framework for hybrid designs.

### Why softmax attention remains dominant at frontier scale (May 2026)

Despite the research results described above, pure Mamba, RWKV, and Hyena models have not displaced softmax transformers as the dominant architecture for frontier-scale models. Several factors contribute:

- **In-context recall.** Global softmax attention enables exact retrieval of information from any position in the context. SSM models maintain a compressed summary state; their recall on tasks that require exact lookups (many-shot learning, long-document QA) is measurably weaker. Hybrid designs recover this.
- **Infrastructure inertia.** The training infrastructure for transformers — FlashAttention, tensor parallelism via FSDP/Megatron, fine-tuning tooling — is highly mature. Switching to a pure SSM requires rebuilding substantial tooling.
- **Hybrid efficiency.** The efficiency gains of SSMs can largely be captured by interleaving a small number of attention layers with many SSM layers (see §ssm-hybrid below), without giving up attention's quality on in-context tasks.

The dominant pattern in frontier labs as of May 2026 is either GQA/MLA transformers for the highest-quality settings, or hybrid SSM-attention models for the highest-throughput settings. Pure SSMs without any attention are a minority in deployed systems.

## Hybrid SSM-attention architectures {#ssm-hybrid}

### Design principle

Hybrid models interleave Mamba (SSM) layers with a small number of full-attention layers. The SSM layers handle the majority of computation at $O(1)$ per-step cost; the attention layers inject global context and support in-context retrieval. A high Mamba:attention ratio (7:1 to 10:1 is common) captures most of the SSM's inference efficiency while the few attention layers preserve quality on tasks requiring long-range recall.

### Jamba (AI21 Labs, 2024)

Jamba (Lieber et al., arXiv:2403.19887, March 2024) interleaves Transformer and Mamba blocks at approximately 1 attention layer per 7 Mamba layers, combined with Mixture-of-Experts (MoE) routing. Total parameters: 52B; active parameters: 12B; context window: 256K tokens. On long-context throughput benchmarks, Jamba achieves 3× the throughput of comparably-sized pure-transformer models. It fits on a single 80 GB GPU.

**Jamba-1.5** (AI21 Labs, September 2024) scales the architecture to 398B total / 94B active parameters across 72 interleaved Mamba+attention layers, using GQA (within attention layers), 16 MoE experts, and ExpertsInt8 quantisation. It extends the context window to 256K tokens. Jamba-1.5 Large is notable as the first non-transformer model scaled to this parameter tier at competitive quality levels.

### Zamba (Zyphra, 2024)

Zamba (Glorioso et al., arXiv:2405.16712, May 2024) is a 7B parameter open-weight model using a Mamba backbone with a single *shared* attention layer — the same attention weight matrix is reused at each position where global context is injected, minimising the parameter overhead of the attention component. Trained on 1T tokens, it is reported as the fastest inference and lowest memory model in the 7B class at long sequences.

### RecurrentGemma and Griffin (Google DeepMind, 2024)

**Griffin** (De et al., arXiv:2402.19427, February 2024) mixes gated linear recurrences with local sliding-window MQA. The recurrence provides efficient state propagation; the local attention windows provide precise local context. Griffin 14B matches Llama-2 on multiple benchmarks while training on 6× fewer tokens, per the paper.

**RecurrentGemma** (Yu et al., arXiv:2404.07839, April 2024) is a production Griffin model: comparable performance to Gemma-2B on standard benchmarks, lower inference latency, higher throughput, $O(1)$ per-step inference memory. Scaled to 9B and 14B.

### Gemma 2 (Google, 2024): attention-hybrid

Gemma 2 (arXiv:2408.00118, August 2024) is not an SSM-hybrid but warrants mention here as an attention-hybrid: it alternates local sliding-window attention ($W=4096$) with global full attention ($W=8192$) every other layer (1:1 ratio), with GQA ($G=2$) throughout. It also uses logit soft-capping ($\tanh(\text{logits}/50) \times 50$) for numerical stability. This design achieves strong benchmark results from a relatively small parameter count.

## Chapter summary

This chapter traced the evolution of the attention mechanism from multi-head attention through MQA, GQA, and MLA, with the KV cache as the unifying constraint. The key relationships:

- **MHA** establishes the baseline: $H$ query heads, $H$ KV heads, $4d$ bytes per token per layer.
- **MQA** reduces to 1 KV head (factor-$H$ cache reduction) at a measurable but moderate quality cost.
- **GQA** introduces $G$ KV heads, recovering near-MHA quality at near-MQA throughput. It is the dominant variant in open-weight frontier models as of 2024–2026.
- **MLA** compresses the KV representation to a low-rank latent, achieving further reductions (3–7× relative to GQA at comparable model scale) through a decompression-and-absorption trick. Used in DeepSeek-V2/V3 as of May 2026.
- **Sliding-window attention** bounds the KV cache by sequence position rather than head count, at the cost of limited receptive field per layer.
- **FlashAttention** is an exact algorithm that reduces HBM I/O from $\Theta(N^2)$ to $\Theta(Nd)$ by tiling the computation in SRAM, making the quadratic forward pass bandwidth-tractable. It does not reduce FLOPs.
- **SSM-hybrid architectures** (Jamba, Zamba, RecurrentGemma) interleave Mamba layers with a small number of attention layers, capturing SSM efficiency while preserving attention's recall quality.

The next chapter, [positional encodings](/textbook/00c-positional-encodings), covers how RoPE interacts with the GQA grouping structure and the MLA decoupled path — completing the picture of how modern attention layers are fully specified.

<Callout variant="exercise" title="Exercise: cache budget arithmetic">
A model has $d=8192$, $H=64$, $L=80$ layers, and uses GQA with $G=8$. You want to serve this model on a node with 640 GB of GPU memory total (8 × 80 GB). The model weights occupy approximately 140 GB in BF16. 

(a) How large is the KV cache per token in bytes?

(b) How many concurrent 128K-token contexts can the node serve simultaneously, assuming 140 GB for weights and the remainder available for KV cache?

(c) If the model switched to GQA with $G=4$, by what factor does the maximum concurrent context count change?

*Answers: (a) $4 \times 8 \times 128 \times 80 = 327{,}680$ bytes $\approx 320$ KB. (b) $(640 - 140) \times 10^9 / (320{,}000 \times 128{,}000) \approx 12.2$, so 12 concurrent contexts. (c) With $G=4$: cache doubles to 640 KB/token; available contexts halve to 6. The factor is 2×.*
</Callout>
